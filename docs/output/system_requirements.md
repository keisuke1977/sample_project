# システム要件定義書

## システム概要

**Femcare（仮）** は、大手企業に勤める女性従業員の健康課題を「気づき→学び→相談」の3ステップで解決するB2B向け女性健康管理Webプラットフォームである。月経・PMS・更年期・妊活など女性特有の体調変化を毎日1分のタップ式チェックインで記録し、パーソナライズされた気づきとコンテンツを提供。看護師・助産師へのチャット相談と産婦人科医とのオンライン相談を組み合わせることで、受診の手前の相談を完結させる。企業・人事側には匿名集計ダッシュボードと健康経営レポートの自動生成を提供し、健康経営優良法人認定の取得を支援する。B2B月額サブスクリプション（¥500〜¥2,000/人・月）で収益化し、1年後に導入企業10社・ARR 1億円を目指す。

### 1. システムの目的

- **解決する具体的な課題：** 女性特有の体調変化（月経・PMS・更年期等）が職場で見過ごされ、不調の蓄積が離職・休職につながっている。婦人科受診のハードルが高く、産業医には「情報漏洩」を恐れて相談できない状況を解消する
- **目標とする成果：** 月次アクティブ率60%以上・専門家相談利用率MAUの20%以上・企業継続率90%以上・NPS（従業員）40以上
- **期待されるビジネスインパクト：** 女性従業員の健康リスクの早期発見と受診促進、離職・休職コストの削減、健康経営優良法人認定の取得支援、女性活躍推進施策のエビデンス提供

### 2. 主な機能

- **コア機能（3〜5個）：** 毎日の体調チェックイン（タップ式・1分以内）、「今日の気づき」パーソナライズフィードバック、看護師・助産師へのチャット相談、産婦人科医とのオンライン相談（テキスト・ビデオ）、ライフステージ別コンテンツ提供
- **差別化機能（2〜3個）：** 企業向け匿名集計ダッシュボード（部署別・年代別体調傾向）、健康経営優良法人認定用データ自動出力、体調データ事前共有による専門家相談の質向上
- **付加価値機能（1〜2個）：** 月次レポートPDF自動生成（経営会議・健保向け）、休職・離職リスクアラート（匿名・集計ベース、Phase 2）

### 3. 提供する価値

- **ユーザーにとっての直接的価値：** 忙しい日常の中でも1分で自分の体調パターンを把握でき、「受診すべきか」を専門家に気軽に確認できる安心感
- **ビジネスにとっての価値：** 健康経営レポートの自動生成による人事工数削減、健康経営優良法人認定取得のエビデンス整備、女性従業員の体調傾向の可視化による施策立案支援
- **社会的価値：** 女性の体調変化に対する職場での理解促進、婦人科受診率の向上、女性活躍推進への貢献

### 4. 実現方法の概要

- **主要なテクノロジー要素：** Next.js（App Router）によるSSR/SSGハイブリッド構成、Supabase（PostgreSQL）によるデータ管理とRow Level Securityによる匿名化・アクセス制御、Clerkによる招待コードベースの認証
- **重要なインテグレーションポイント：** Web Push通知（チェックインリマインダー・返信通知）、ビデオ通話SDK（産婦人科医オンライン相談）、PDF生成ライブラリ（月次レポート）
- **スケーラビリティへの考慮：** MVP段階は数社・数百人規模。Vercelのサーバーレス構成とSupabaseのマネージドPostgreSQLにより、1年後の数十社・数万人規模への拡張をインフラ変更なしに対応

## 非機能要件

### プライバシー・セキュリティ（最重要）

| 要件 | 詳細 |
|------|------|
| 個人特定防止 | 企業・人事側から個人の体調データを特定できない設計。集計は5名以上のグループ単位。Row Level Securityによるデータアクセス制御 |
| データ暗号化 | 通信はすべてTLS 1.3以上。データベース内の医療関連情報は保存時暗号化（AES-256相当） |
| 認証・認可 | Clerkによる多要素認証対応。招待コード制による閉域アクセス。企業管理者・従業員の権限分離 |
| 同意管理 | 初回登録時のプライバシーポリシー同意取得。同意内容の版管理・記録 |
| 相談プライバシー | 相談内容は当該従業員のみ閲覧可能。企業側への非公開を技術的に保証 |
| GDPR/個人情報保護法 | 個人情報保護法に準拠したデータ取得・管理・削除の実装 |

### パフォーマンス

初期ユーザー規模として数百人（パイロット段階）を想定し、1年後の数万人規模を見据えた設計とする。

| 要件 | 目標値 | 根拠 |
|------|--------|------|
| チェックイン画面の表示 | 1秒以内 | 毎朝の習慣化に直結するため遅延は継続率に影響 |
| 相談メッセージの送信 | 即時（非同期処理） | ユーザーの体感として遅延なく送信完了を確認できること |
| ダッシュボードデータ反映 | リアルタイム〜24時間以内 | 人事担当者は月次利用が主のため24時間は許容範囲 |
| 同時接続数 | MVP: 100同時接続、1年後: 5,000同時接続に耐えられる設計 | パイロット300名・1年後数万人規模から推定 |
| 可用性 | 99.5%以上（月次ダウンタイム約3.6時間以内） | 朝のチェックインが主な利用のため深夜メンテナンスで対応可能 |

### スケーラビリティ

- **MVP段階：** 1〜3社・数百名規模（Vercelサーバーレス＋Supabase無料〜Proプランで対応）
- **1年後：** 10社・数万名規模（Supabase Businessプランへの移行、DB接続プーリング設定）
- **将来：** 水平スケール設計を前提とし、ステートレスなAPI設計を徹底

### 医療規制対応

| 要件 | 詳細 |
|------|------|
| 医師法・薬機法の遵守 | 「診断」「処方」に相当する表現・機能は実装しない。「相談」と「診療」の線引きを機能設計・コンテンツガイドラインに明文化 |
| 受診促進 | 専門家相談は「受診を促す」目的に留め、医療行為とならない設計 |
| コンテンツ監修 | 医師・専門家の監修のもとコンテンツを制作・定期的に更新 |
| 免責事項 | 各相談・コンテンツ画面に「このサービスは医療行為ではありません」を明示 |

## 技術スタック

### 推奨スタック

| レイヤー | 技術 | 選定理由 |
|----------|------|----------|
| フロントエンド | **Next.js 14+ (React)** | SSR/SSGハイブリッドによる初期表示高速化。App RouterによるServer Componentsで体調グラフ等のデータ表示を最適化。モバイルファーストのUIをReactコンポーネントで効率的に構築 |
| 言語 | **TypeScript** | 医療データを扱う上での型安全性の確保。チーム開発での保守性向上。APIレスポンスの型定義による信頼性担保 |
| バックエンド | **Next.js App Router (API Routes)** | フロントエンドと同一リポジトリで管理。サーバーレス関数として動作し、スモールスタートのコスト効率が高い |
| データベース | **Supabase (PostgreSQL)** | Row Level Securityによる行レベルのアクセス制御が匿名化要件に直結。Realtime機能でチャット相談のリアルタイム更新を実現。マネージドサービスによる運用コスト削減 |
| 認証 | **Clerk** | 招待コードベースのユーザー管理に対応。多要素認証・組織管理機能が標準搭載。Supabaseとの統合が容易 |
| ホスティング | **Vercel** | Next.jsとの親和性が最高。エッジキャッシュによる国内ユーザーへの高速配信。プレビューデプロイによる開発効率向上 |
| ビデオ通話 | **Twilio Video / Daily.co** | 産婦人科医とのビデオ相談に使用。マネージドSDKによる開発コスト削減。HIPAA準拠オプションあり |
| PDF生成 | **React PDF / Puppeteer** | 月次レポート・健康経営認定用データのPDF出力に使用 |
| Push通知 | **Web Push API + Service Worker** | チェックインリマインダー・相談返信通知。ネイティブアプリなしでのPush通知を実現 |
| メール通知 | **Resend / SendGrid** | Push通知の補完。チェックインリマインダー・返信通知のメール送信 |
| モニタリング | **Sentry + Vercel Analytics** | エラートラッキングとパフォーマンス監視 |

### 採用技術の補足評価

**代替案との比較：**
- **BaaS（Firebase等）vs Supabase：** Supabaseを選択。PostgreSQLのRow Level Securityが個人特定防止の要件に技術的に合致する点が決定的な差別化要因。また、PostgreSQLの成熟したエコシステムがスケール時のリスクを低減
- **Next.js vs Remix：** Next.jsを選択。エコシステムの成熟度とVercelとの統合による開発効率を優先
- **Clerk vs NextAuth：** Clerkを選択。招待コードベースのB2Bユーザー管理、組織・権限管理機能が標準搭載されており、開発工数を大幅に削減

## 開発ロードマップ

### フェーズ分けの考え方

MVPでは最小限の仮説検証機能に絞り、パイロット企業3社でユーザー行動・ニーズを検証してからPhase 2以降の機能追加判断を行う。リスクが高い機能（ビデオ通話・PDF生成）を早期に実装することで技術的リスクを前倒しで解消する。

### Phase 1: MVP開発（1〜3ヶ月目）

**目標：** パイロット導入可能なMVPをリリース

| 期間 | 作業内容 |
|------|----------|
| 1ヶ月目 | インフラ構築（Supabase・Vercel・Clerk）、認証・オンボーディング、体調チェックイン機能 |
| 2ヶ月目 | コンテンツ閲覧機能、チャット相談（看護師・助産師）、通知機能（Web Push・メール） |
| 3ヶ月目 | 産婦人科医オンライン相談（テキスト・ビデオ）、管理者ダッシュボード基本機能、PDF出力、健康経営認定データ出力 |

**リリース判定基準：** チェックインの1分以内完了・チャット相談の送受信・ダッシュボードの匿名集計表示・PDF出力が動作すること

### Phase 2: パイロット〜正式リリース（3〜9ヶ月目）

**目標：** パイロット3社・300名でのユーザー検証と改善、正式リリース

| 期間 | 作業内容 |
|------|----------|
| 3〜6ヶ月目 | パイロット運用・週次インタビュー・UX改善。体調カレンダー・グラフ、月経周期トラッキング追加 |
| 6〜9ヶ月目 | パイロット改善版リリース（正式版）。休職・離職リスクアラート、症状トレンド、体調データ事前共有、短動画コンテンツ追加 |

### Phase 3: 成長期（9〜12ヶ月目以降）

**目標：** 導入企業10社・ARR 1億円

| 期間 | 作業内容 |
|------|----------|
| 9〜12ヶ月目 | 健保連携交渉・健保提出用レポート対応。産業医連携機能の設計着手。コンテンツ検索・カスタムレポート機能 |
| 1年以降 | 産業医連携機能実装。ネイティブアプリ（iOS/Android）検討。多言語対応・海外展開検討 |

| ビデオ通話の品質・安定性 | 高 | マネージドSDK（Twilio/Daily.co）の採用。MVP段階ではテキスト相談を主軸にビデオは補完的位置づけ |
| データ匿名化の不完全性 | 高 | Supabase RLSによる技術的制御＋5名未満グループの非表示ルールを設計初期から実装。第三者セキュリティレビューを実施 |

| スケールアップ時のDB性能 | 中 | Supabase接続プーリング設定・インデックス最適化・集計処理のバッチ化 |

## UI/UXの方向性

- **デザインコンセプト：** 「やさしい、プロフェッショナル」。医療テーマの清潔感・信頼感を保ちながら、女性が毎日使いたくなる温かみのある雰囲気。「病院のような堅さ」は避ける
- **トーン＆マナー：** 寄り添い・安心感・シンプル。医療用語より平易な言葉を使用（「黄体期」より「月経前の時期」）。過度な医療感（注射・赤十字等のアイコン）は使用しない
- **ターゲットデバイス：** モバイルファースト（従業員は通勤中・昼休みのスマホ利用が主）＋PCレスポンシブ対応（人事担当者はPC利用が主）
- **カラー：** Primary: ダスティローズ（`#C97A72`）、Background: ウォームホワイト（`#FAF8F5`）、Accent: セージグリーン（`#4A7C6F`）
- **フォント：** Noto Sans JP（日本語）+ Inter（英数字）。本文15px/行間1.75
- **参考サービス：**
  - [Notion](https://www.notion.so) - シンプルで整理されたレイアウト・余白の使い方
  - [Flo（月経管理アプリ）](https://flo.health) - 女性健康アプリの情報設計・グラフ表示
  - [Calm](https://www.calm.com) - 体験の温かみ・通知フィードバックのトーン
